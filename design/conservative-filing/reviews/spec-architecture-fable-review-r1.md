# Conservative-filing SPEC — architecture lens review r1 — NOT GREEN (1 Critical + 6 Important)

_Fable, independent, general-engineering/architecture lens (tax correctness reviewed separately). Verified
the SPEC's claims about existing machinery against current source._

---

VERDICT: NOT GREEN

Verified positives first (claims that held): the P2 HIFO comparator claim is TRUE (`hifo_cmp` sorts `usd_basis == 0` lots last, `crates/btctax-core/src/project/pools.rs:272-287`); D-3 is TRUE (`ForbiddenBroker2027` fires from the pure custody envelope `is_broker` + `persistability`, `crates/btctax-core/src/optimize.rs:453-488` — no transfer-statement modeling needed); D-5 is TRUE (`pseudo_active()` = `pseudo_synthetic_count > 0`, set only by synthetic pseudo-mode decisions, `crates/btctax-core/src/state.rs:273-284`; `Lot.pseudo` is "always false outside pseudo mode", state.rs:125-131 — a persisted conservative `Acquire` cannot trip the export gate); P5's data assumption is TRUE (5,801 data rows, 2010-07-17→2026-06-03, `crates/btctax-adapters/data/btc_usd_daily_close.csv`, `BundledPrices` provider at `crates/btctax-adapters/src/price.rs:10-27`; a day-iterating `window_low` over `&dyn PriceProvider` is trivially buildable); P6's engine-reuse claim is credible via the existing clone-fold-discard what-if seam (`crates/btctax-core/src/whatif.rs:1-30`, `optimize::synthetic_state`); P3/P8 have a clean home (advisory `Blocker` severity exists, state.rs:20,80-102, and `Consumed` carries `basis_source` to the leg, pools.rs:289-298). The findings below are what remains.

1. [Critical] D-1 / P1 / §3 (whole design) — The 2025 per-wallet transition destroys the tranche tag (Path A) or discards the tranche entirely (Path B), and the SPEC never mentions the transition. Every realistic tranche is pre-2025-dated (the persona is "held since before 2016"; D-2 sets `acquired_at` = window end), so it folds into `PoolKey::Universal` (`pools.rs:15-21`). At the 2025-01-01 seed, Path A (the default when no allocation is effective, `resolve.rs:1292-1294`) OVERWRITES every surviving lot's `basis_source` to `ReconstructedPerWallet` (`crates/btctax-core/src/project/transition.rs:82-86`); Path B DISCARDS the Universal remainder and seeds only the allocation's lots (`transition.rs:76-77, 88-94`). Worse, declaring a new pre-2025 tranche in a vault with an existing effective `SafeHarborAllocation` changes the Universal residue, so the conservation guard hard-blocks it (`SafeHarborUnconservable`, `resolve.rs:1227-1245`) and the allocation goes inert → Path A → overwrite. Net effect: for every 2025+ disposal — i.e. every disposal this 2026-built feature will ever serve — `EstimatedConservative` never reaches the disposal leg, so P3 (dip advisory), P7 (mandatory disclosure trigger), P8 (custody nudge), and D-6's 8949 treatment are keyed on a tag that no longer exists, and P1's claim "`basis_source` propagates to the disposal leg and the 8949 row" is false. (P2's steering alone survives, since `hifo_cmp` keys on `usd_basis`, not the source.) Fix: specify the transition interaction explicitly — exempt `EstimatedConservative` from the Path-A overwrite (preserve the tag through the seed), define the Path-B story (forbid pre-2025 tranche declaration in a Path-B vault, or require an amended allocation covering the tranche sats), and add a tranche-through-transition KAT to §6.

2. [Important] D-6 — The claim "this E-vs-F split is ALREADY driven by `forms.rs`'s Exchange-vs-SelfCustody logic (`box_needs_review`) — P1 reuses it, adds no new box logic" is FALSE. `Form8949Box` has exactly two variants, C and F (`crates/btctax-core/src/forms.rs:34-40`), and the module doc explicitly says "We NEVER auto-assign A/B (ST) or D/E (LT)" (forms.rs:29-33). `box_needs_review` is a bool review FLAG (forms.rs:54, 121), not a box assignment. Emitting Box E for Exchange tranche disposals is new work: an enum variant, assignment logic, and CSV/PDF plumbing. Fix: either re-scope D-6 to the existing conservative posture (F + `box_needs_review`, defer E) or explicitly scope the new box logic into P1.

3. [Important] D-6 / P1 — "date acquired = VARIOUS" has no representation path. `Form8949Row.date_acquired` is a typed `TaxDate` (forms.rs:58-59), rendered via `fmt_date` in the PDF fillers (`crates/btctax-forms/src/fill8949.rs:38`, `fill8949_full.rs:49`) and `.to_string()` in the CSVs (`crates/btctax-cli/src/render.rs:1047, 1102`). Supporting VARIOUS requires a row-type change (date-or-various) plus every renderer plus a CSV-schema decision — none scoped. Fix: scope that work in P1, or drop VARIOUS and print the window-end date the row already carries (also i8949-compliant for a single-row tranche).

4. [Important] D-1 vs P3/P5/P6/P7 — the asserted window `[start, end]` has no persisted home, contradicting "the new `BasisSource` variant is the sole schema addition." `Acquire` is `{sat, usd_cost, fee_usd, basis_source}` (`crates/btctax-core/src/event.rs:48-54`) and `LedgerEvent` adds only id/timestamp/tz/wallet (event.rs:350-358). `acquired_at` = window END survives as the event timestamp; the window START is lost — and since v1 files $0 (D-7), the floor is not recoverable from `usd_cost` either. But P5/P6 must compute `window_low` at report time, P7 must enumerate "each tranche's window", and P3 names "your <window> tranche". None of that is computable from the tagged `Acquire` alone. Fix: add an optional `#[serde(default)]` window field to `Acquire` (or a companion declaration payload — see finding 5) and amend D-1.

5. [Important] P1 (input path) — no identity scheme exists for a user-declared `Acquire`, so "a CLI verb … emits the tagged `Acquire`" is not buildable as specified. `EventId::Import` requires one of exactly four venue `Source`s (Swan/Coinbase/Gemini/River — `crates/btctax-core/src/identity.rs:8-13, 56-62`); there is no Manual source, all production `Acquire`s come from the four adapters, and tui-edit's authoring paths mint only Decision-id payloads. Wrapping an `Acquire` in a `Decision` id is an untested hybrid: `is_imported()` gates primary-movement semantics (event.rs:333-345) and the fingerprint/conflict/supersede machinery assumes Import ids. Fix: specify the identity scheme — either a `Source::Manual`-style variant (with its `priority()`/`tag()`/persistence implications) or a first-class `DeclareTranche` decision payload that folds like an acquire, which would also give the window from finding 4 a natural home.

6. [Important] P2 — steering holds only under HIFO, and the SPEC specifies no mechanism that makes HIFO effective. FIFO is the engine default absent an election (event.rs:240-243), and `method_order` FIFO is acquisition-date ASC (pools.rs:255-260) — under which an old-era $0 tranche is consumed FIRST, the exact inversion of steering, silently maximizing gain (P3 would fire, but the "documented lots first" guarantee is gone). The SPEC's "HIFO (the posture default)" names no mechanism: P1's flow emits no `MethodElection`, and elections are global-or-Exchange-scoped with `effective_from ≥ TRANSITION_DATE` (event.rs:240-258); pre-2025 disposals use `config.pre2025_method` (default FIFO). Fix: specify how the posture is established (declare-tranche emits/requires a HIFO election, or at minimum an advisory when the in-force method would consume a tranche while documented lots remain), and state the method dependence in P2.

7. [Important] D-7 vs D-1/D-6/P1 — internal contradiction on the floor. D-7 mandates "no `> $0` value is ever written to a filed 8949 by the conservative flow itself" and keeps floor-promotion in Approach B; yet D-1 specifies `usd_cost = <floor or 0>`, P1's input offers "window-low floor" as a declarable basis position, and D-6 says col (e) = "the conservative $0/floor". A floor-basis lot's basis flows unconditionally to the 8949 (`cost_basis: leg.basis`, forms.rs:126) — there is no mechanism to file a different basis than the lot's. Fix: make v1 accept only $0 declarations (align D-1/D-6/P1 wording; P5's floor then feeds only P6's informational delta), or renounce the D-7 mandate — one or the other, before planning.

8. [Minor] D-1 / P1 — "the 8949 'how acquired' column" does not exist; `how_acquired_from` (forms.rs:236-248) is the Form 8283 "How acquired by donor" mapping for donations. The new variant WILL compile-force that match arm, and the SPEC should decide the 8283 category for a donated tranche coin (Review seems right) and correct the form name.

9. [Minor] P1 — the input list ("quantity, window, basis position") omits the tranche's WALLET, which the fold hard-requires (`"acquire without wallet"` blocker, `crates/btctax-core/src/project/fold.rs:568-577`) and which is load-bearing for P4/P8 and the Path-A transition destination.

10. [Minor] P5 — partial window overlap with the dataset is unspecified: the data starts 2010-07-17, and the persona can plausibly assert a 2009–2011 window; "None if the window is outside the dataset" does not cover partial overlap, and min-over-the-covered-part would overstate the floor vs the true low (informational-only in v1, hence Minor).

11. [Minor] D-1 — the exhaustive-match sweep is understated: besides forms.rs:239, the new variant compile-forces `crates/btctax-cli/src/render.rs:44-53` (CSV label) and `crates/btctax-tui-edit/src/edit/form.rs:1756-1780` (edit-cycle ring + label; decide on-ring vs off-ring, precedent: `SelfTransferInbound` is off-ring). Contained, compiler-enforced changes — but the "sole schema addition" framing hides them from the plan.

12. [Minor] P6 — timing semantics are fuzzy: "delta between its filed $0 position and the window-low floor position" implies tranche legs already consumed in a filed year, but the G-3 lever is most valuable BEFORE selling into the tranche. Specify which disposals/years the delta covers (year-scoped filed legs vs hypothetical future consumption) and where the nudge surfaces (report / optimize / TUI).

13. [Nit] P1 — a vault containing the new `BasisSource` variant fails to load on older binaries (the documented forward-only trade-off; precedent event.rs:270-274). Worth one line in the SPEC; harmless in practice (no installed base).

Summary: the codebase anchors the SPEC cites are mostly real and accurately located (pools.rs comparator, optimize.rs envelope, pseudo gate, price data), but two D-6 "already exists" claims are false (Box E, VARIOUS), D-1's "sole schema addition" is false on three axes (window state, event identity, filed-floor contradiction), P2's emergent steering is method-gated with no mechanism specified — and the one genuinely fatal omission is that the 2025 Universal-pool transition erases the `EstimatedConservative` tag before any disposal the feature will ever touch, which invalidates P1's propagation claim and everything keyed on it. The design is salvageable with contained changes (a transition exemption, a declaration payload carrying the window, $0-only v1), but not buildable as written.
