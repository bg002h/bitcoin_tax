## Findings

### IMPORTANT

**I-1 ŌĆö Task 2 (D-1a-b / arch r3 N-4): the numeric `decision_seq` tie-break is dead code; the plan reintroduces the exact lexicographic misorder the SPEC Ōśģ-pinned against.**
The plan sets `src_ref: SourceRef::new(&format!("{seq}"))` and appends `.then(decision_seq(&a.id).cmp(...))` as the FOURTH sort key. But `SourceRef` derives `Ord` on its inner `String` (`identity.rs:34-36`) and `sort_canonical` compares `src_ref` at key 3 (`resolve.rs:1381`) ŌĆö two same-window tranches (same utc = midnight window_end, same `src_priority = u8::MAX`) have DIFFERENT `src_ref`s ("2" vs "10"), so the string compare decides and the numeric key is never reached: seq 10 folds before seq 2. That is verbatim the r3 Nit-4 failure mode ("If routed through sort_canonical's src_ref (string-ish), lexicographic compare misorders seq 2 vs 10"), which SPEC D-1a-b forbids; the plan's own comment ("display only; ties break numerically below") is false about its own mechanism, and no planned KAT would catch it.
**Fix:** give tranche Effs a constant `src_ref` (e.g. `SourceRef::new("")`) so ties fall through to the numeric key ŌĆö or simpler, make the final key `.then(a.id.cmp(&b.id))` (`EventId::Decision{seq}`'s derived `Ord` compares `u64` numerically; identical ids never both appear). Add a KAT: two same-window tranches with seqs 2 and 10 fold in seq order (this also pins D-1a-d's "two tranches are legitimately additive", currently untested ŌĆö see N-3).

**I-2 ŌĆö Tasks 1/3 (D-1a-d): missing sweep site ŌĆö `is_revocable_payload` (`crates/btctax-core/src/void.rs:20-34`) will silently classify a `DeclareTranche` as NON-revocable on every void surface.**
The predicate is a non-exhaustive `matches!` list (not compile-forced). Without adding `EventPayload::DeclareTranche`, `voidable_decisions` ŌĆö documented as "the SINGLE shared source of truth for which reconcile decisions may be swept-voided" (`void.rs:1-5`) ŌĆö excludes tranches from the bulk-void plan (`session.rs:1149`) and both TUI void flows (`tui-edit/main.rs:3861`, `:8237`), and the TUI's revocability messaging treats it as permanent (`editor.rs:232-239`). The engine-side void does apply (pass-1a `Some(_)` catch-all, `resolve.rs:464-466`, and the single CLI `void`'s `guard_decision_conflict` passes it), so Task 3's KAT goes green while the product's primary void surfaces contradict D-1a-d ("voidable via the generic `VoidDecisionEvent`"). The plan's sweep covers only the four `BasisSource` matches + `is_imported`; this `EventPayload` site is nowhere in the plan.
**Fix:** add `EventPayload::DeclareTranche(_)` to `is_revocable_payload` in Task 1, plus a KAT (`voidable_decisions` over a ledger with a tranche lists it). Cosmetic sibling: `bulk_void_payload_summary` (`btctax-cli/src/main.rs:2085-2133`) falls to a `Debug` wildcard ŌĆö add a readable arm.

**I-3 ŌĆö Task 6 (D-8 both-directions refusal): the allocation-append site inventory is wrong, and the allocation-authoring residue will list tranche lots as allocatable.**
The plan's Files say "wherever `EventPayload::SafeHarborAllocation` is appended **in `btctax-cli`**". Actual append sites: `btctax-cli/src/cmd/reconcile.rs:984` (allocate) and `:1273` (attest) **plus** `btctax-tui-edit/src/edit/persist.rs:1031` (TUI allocate) and `:1114` (TUI attest). Guarding only the CLI leaves the TUI able to record an allocation while a pre-2025 tranche exists ŌĆö the SPEC-mandated record-time refusal ("fires in BOTH directions") silently holds on only one of the two record surfaces (the Task-5 backstop still denies effectiveness, so no silent discard ŌĆö but the SPEC's friendly-error layer is half-missing and the planned KATs only cover the CLI). Compounding it: `Session::safe_harbor_residue` (`btctax-cli/src/session.rs:681-700`) keeps ALL decisions in its pre-2025 projection (`_ => !matches!(SafeHarborAllocation)`), so once `DeclareTranche` folds lots, the allocate opener (shared by CLI + TUI) will pre-populate the tranche's `$0` sats as allocatable pre-2025 residue ŌĆö authoring exactly the coexistence allocation v1 forbids (and, for a Ōēź2025-window tranche, listing a lot that isn't even pre-2025 residue, producing a guaranteed-unconservable allocation dead-end).
**Fix:** name all four append sites (or route the guard through one shared chokepoint each in `session`/`persist`); extend the Task-6 KATs to the TUI persist path; and in `safe_harbor_residue`, either exclude `DeclareTranche` decisions from the residue projection or refuse opening the allocate flow when a tranche exists (the same in-force predicate).
