## Findings

**[Important] TUI void flows render a DeclareTranche as `"?"` / `"?"` — the taxonomy sweep missed `summarize_void_payload`** — `crates/btctax-tui-edit/src/main.rs:3830` (wildcard `_ => ("?", "?".to_string(), None, false)`), consumed by BOTH `open_void_flow` (`main.rs:3861`) and `open_bulk_void_flow` (`main.rs:8245`). Scenario: user declares a tranche, later hits the D-8 refusal whose own message says "Void the tranche first", opens the TUI void flow — the tranche row shows tag `?`, summary `?`. Task 1 deliberately added `DeclareTranche` to `is_revocable_payload` *so that* "bulk-void + both TUI void flows" can void it, and added a readable arm to the CLI sibling (`bulk_void_payload_summary`, `btctax-cli/src/main.rs:2154`), but the TUI's renderer for the exact same `voidable_decisions` rows was missed. This is the precise "whole-surface sweep on taxonomy change" failure mode this project has already codified as a lesson, on a safety-relevant selection surface (the user is choosing what to void from an unlabeled row). Fix: add an arm `("DeclareTranche", format!("{} sat in … (window {}..{})"), None, false)` mirroring the CLI summary, plus a `summarize_void_payload` human-label test (the `…classify_inbound_is_human` precedent exists at `main.rs:9927`).

**[Minor] `in_force_allocation_exists` diverges from engine void semantics for a dangling void on an EFFECTIVE allocation — friendly layer misses; backstop holds** — `crates/btctax-cli/src/cmd/tranche.rs:31-53`. The record-time "voided" view counts ANY `VoidDecisionEvent` naming the id. The engine does not: an allocation void goes to the separate `allocation_voids` bucket (`resolve.rs:466-472`) and a void of an *effective* allocation is adjudicated inert — the allocation **stays in force** (`resolve.rs:1344-1352`). The single CLI `void` verb performs **no** `effective_alloc` check (documented at `reconcile.rs:822`), so this state is product-reachable: allocate → effective → `reconcile void decision|N` (→ Hard DecisionConflict, allocation still effective) → `declare-tranche` with a pre-2025 window now records CLEANLY beside a still-effective allocation. Consequence: no silent discard (the T5 backstop denies the allocation effectiveness on the next projection, loudly), but the friendly refusal the SPEC promises for "ANY in-force allocation" doesn't fire, and the filer's pre-2025 regime flips Path B → Path A via a blocker storm instead of a refusal. Note the implementation faithfully matches the plan's own predicate definition ("not in `voided`") — the divergence is inherited from the plan; the vault was also already hard-blocked by the bad void. Suggested fix: on the allocation side of the predicate, count a void only if the allocation is engine-inert (mirror `voidable_decisions`' blocker-keyed `effective_alloc` test), or simply refuse the tranche when any non-voided-or-effective allocation exists.

**[Minor] Both ATTEST-site guards are untested — a mutation deleting either survives the suite** — `reconcile.rs:1199` and `persist.rs:1105`. Task 6's tests cover CLI allocate, TUI allocate, and the tranche side; neither attest guard has a discriminating test. They are not pure dead code: the finding-2 corner (dangling void + tranche) makes the attest guard the live defense on the re-append path. Per this repo's own mutation discipline (`untested-guard-pattern`), add a fixture reaching each attest guard (the dangling-void recipe above reaches it through the product), or at minimum a unit test that `guard_allocation_vs_tranche` is invoked on the attest paths.

**[Minor] `safe_harbor_residue`'s tranche exclusion overs sells the throwaway projection in the feature's central scenario — skewed opener display (fail-closed, but garbage)** — `crates/btctax-cli/src/session.rs:687-700`. The filter drops the `DeclareTranche` event but keeps pre-2025 disposal imports. For the canonical conservative filer — pre-2025 disposals covered only by the tranche (e.g., tranche 1 BTC + documented buy 0.2 BTC + 2020 sale 0.5 BTC) — the filtered projection has a 0.3 BTC shortfall: the fold emits an internal "dispose short by N sat" blocker (ignored) and the sale eats *documented* lots the tranche actually covered, so the TUI allocate opener displays an understated/empty allocatable residue. Nothing wrong can be recorded (CLI guards before residue; TUI persist refuses), and the new KAT (`safe_harbor_residue_omits_tranche_sats`) only exercises the no-disposal case, so this skew is untested and unnoticed. The plan explicitly offered the alternative ("refuse opening the allocate flow when a tranche exists") which avoids the garbage display; either adopt it for the pre-2025-tranche case or pin the disposal-present skew as accepted behavior with a comment/test.

**[Minor] The backstop KAT does not exercise the SPEC's named ordering** — `kat_tranche.rs:533-564` fixes tranche = seq 1, allocation = seq 2, i.e. declare-then-allocate; SPEC D-8 demands the KAT pin the **inert-then-declare** ordering (allocation recorded first) that r3-New-1 found. I verified order-independence is structural (the effectiveness loop computes `universal_snapshot` per allocation from the complete timeline; seq order never enters), so this is a test-shape gap, not a behavior gap — but it's a one-line twin (swap the seqs) and the SPEC text names it explicitly.

**[Nit] Engine-level `DeclareTranche` accepts `sat ≤ 0` / inverted windows** — validation lives only in the CLI record path (`tranche.rs:110-123`); a library caller or hand-crafted vault folds a negative-sat $0 lot and corrupts `sigma_in`. Consistent with the existing posture for imported payloads, but the admit's own comments treat hand-crafted vaults as a soft threat (the id-guard, the `applied` bypass) — a cheap resolve-time blocker for `sat ≤ 0` would make the posture uniform.

**[Nit] The KAT-G1 workaround is legitimate but precedent-setting; house the predicate outside `cmd::`** — `btctax-cli/src/lib.rs:22` re-exports `cmd::tranche::guard_allocation_vs_tranche` so `persist.rs:1032/1105` can call it without the forbidden `cmd::` token. I read the gate (`persist.rs:1896-2003`): its rationale is session-lifecycle/lock hazards, and this guard is a pure `&[LedgerEvent] → Result` predicate — no session, no lock, no I/O — so the *intent* is honored, not evaded. But the mechanism (root re-export of a `cmd::`-homed fn) is a template a future session-opening fn could copy. Move the predicate to a non-`cmd` module (`session.rs` or a `guards` module) or add a comment at the re-export stating the pure-predicate exemption rationale.
