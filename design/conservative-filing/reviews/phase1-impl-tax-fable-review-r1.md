## Findings

**[Minor] `build_op`'s DeclareTranche arm is reachable from the import fold path — a smuggled payload now folds a real lot where it used to fold `Op::Skip`** — `crates/btctax-core/src/project/resolve.rs:396` (the arm), `:1109` (`applied.get(&e.id).unwrap_or(&e.payload)`), `:584` (pass-1c inserts `as_` with **no payload-type validation**). Two hand-crafted-vault vectors: (a) a `ClassifyRaw{target: <import>, as_: DeclareTranche{…}}`, and (b) a raw event with an `EventId::Import` id and a `DeclareTranche` payload (the `:1074` Decision-id guard correctly rejects it from the admit, but it then falls through to the import loop, which reads `&e.payload` and calls `build_op`). Either folds a `$0 EstimatedConservative` lot homed at the **import timestamp, not `window_end`** — bypassing D-2 homing and both record-time guards. Mitigations verified: the CLI `classify-raw` verb refuses non-`is_imported()` payloads at record time (`reconcile.rs:467-470`), so no product surface can author this; a hand-crafter can forge worse directly (an arbitrary-basis `Acquire`); and the projection backstop still fires on the tagged lot if it lands pre-2025. But note the delta this feature introduced: before this branch, any decision payload smuggled through `applied` hit `_ => Op::Skip` (inert); now `DeclareTranche` is the first decision payload with primary-movement effect through that path — exactly the "forge one" half of the arch-r2 NEW-N-2 concern, closed on the decision side but not the import side. **Fix:** guard the arm on the id (`EventId::Decision{..}` ⇒ `Op::Acquire`, else `Op::Skip` — the no-Skip KAT pins only the legitimate path), or reject decision payloads in pass-1c with a `DecisionConflict`.

**[Minor] The backstop's non-poisoning direction has no projection-level KAT** — `crates/btctax-core/tests/kat_tranche.rs` (absent), `crates/btctax-cli/tests/declare_tranche_cli.rs:532-553` (test (d) checks **record acceptance only**). I verified by code-trace that a `window_end ≥ 2025-01-01` tranche cannot poison an allocation (its Eff date fails `universal_snapshot`'s `e.date() < TRANSITION_DATE` filter at `transition.rs:51`, and `pool_key` routes it per-wallet, so `estimated_conservative_remaining_sat` stays 0), and existing effective-allocation tests would go red if the backstop over-fired on tranche-free vaults. But no test pins "≥2025 tranche + conserving allocation ⇒ no `SafeHarborUnconservable`, transition is `PathB`". A future regression (e.g., someone re-keys the sum on payload presence instead of pool residue) would silently flip every mixed-records filer from Path B to Path A — never an understatement, but it strips Rev-Proc 2024-28 flexibility and defeats the exact foreclosure guard tax r1 I-2 demanded. **Fix:** one KAT — pre-2025 documented buy + conserving attested allocation + a ≥2025 tranche → assert no `SafeHarborUnconservable` and `TransitionMode::PathB`.

**[Minor] Both ATTEST-site guards are untested — their mutation survives** — `crates/btctax-cli/src/cmd/reconcile.rs:1199` and `crates/btctax-tui-edit/src/edit/persist.rs:1105`. Deleting either guard leaves the suite green: the CLI tests cover the tranche path (a/a2), CLI allocate (c), and TUI allocate (`persist.rs:5042` test); nothing exercises either attest guard. They are genuine defense-in-depth (I traced the reachable orderings: allocate→void→declare-pre-2025-tranche→attest is the only route, and without the guard, attest still fails on "no live allocation"), but the project's own untested-guard discipline and Task 6's "all FOUR append sites" framing imply each wired site should kill its mutation. **Fix:** stage allocate→void-allocation→declare-tranche→attest in both CLI and TUI tests and assert the refusal is the *tranche* message (which discriminates the guard from the no-allocation error).

**[Minor] `safe_harbor_residue`'s shadow projection mis-states documented remainders when pre-2025 disposals actually consumed tranche sats** — `crates/btctax-cli/src/session.rs:695-698`. Excluding the `DeclareTranche` event from the residue projection is right for the self-poisoning and ≥2025-leak cases (comment is accurate), but a tranche is a *primary movement*, unlike every other retained decision: if real pre-2025 history drew disposals against tranche sats (FIFO, tranche `window_end` earliest), the shadow projection re-attributes those draws to documented lots (or goes negative), so the TUI allocate opener can display documented residue lots with understated remaining sat/basis. Fail-closed downstream — any allocation authored from it is refused at all record sites and unconservable at projection — so no tax result is wrong; it is a reachable incorrect *display*. **Fix:** the plan's own alternative — refuse opening the allocate flow when a pre-2025 tranche exists — or document the distortion at the call sites.

**[Nit] The allocation-side refusal reuses a hint worded for the opposite direction** — `crates/btctax-cli/src/cmd/tranche.rs:68-72`: "…Void the tranche first, or revisit the in-app safe-harbor allocation; if your filed allocation is already final…" — the user in this direction is *creating* the in-app allocation, so "revisit the in-app safe-harbor allocation" only parses if read as the real-world-filed one. The hedge itself is correct, provenance-neutral, and no message overstates authority or gives legal advice (all strings checked, including the backstop blocker detail at `resolve.rs:1297-1301` and the future-window warning).
