## Important

### r4-I1 — the provenance class is OTS-only, so the **taxcalc-side** printed-vs-exact-operand residual above the Table ceiling has no absorbing class; the §5.1-mandated high-income SE cells make the resulting undeclared red an expected event

**Anchors:** spec §6.2(b) (the predicate is declared for "the oracle" but the class tuple is `(OTS, …)`), §6.4 second class bullet; §5.1 axes (W-2 high >$250k; SE over the $250k threshold; the t=3 triple); `crates/btctax-core/src/tax/method.rs:47-56` (above `TAX_TABLE_CEILING` every operand takes the TCW — the taxcalc Table class is then *correctly* false); `crates/btctax-core/tests/goldens/full_return_goldens.json` (`mfj_se_over_the_addl_medicare_threshold`: OTS TI 253,942.94, OTS L16 47,031.31, taxcalc L16 47,031.318 — an above-ceiling, cents-carrying household); `crates/btctax-core/tests/golden_returns.rs:94-213` (no declared taxcalc divergence above the ceiling today).

**Failure.** For a household where **no** worksheet operand is below the ceiling (e.g. the §5.1-mandated SE-over-$250k / high-W-2 cells: standard deduction, no preferential income — `regular_tax` → pure TCW), part (b) compares `Table_btctax(reproduced printed operands)` = TCW on the whole-dollar printed TI against `round_dollar(taxcalc L16)` = taxcalc's exact schedule on its exact-cents TI. The two differ by `rate × δ` where δ = printed-TI − exact-TI ≠ 0 for every cents-carrying household (SE half-deduction and QBI put cents into exact TI; taxcalc's own exact figures even differ from OTS's in the mills — JSON: 70,008.908 vs 70,008.94). The rounded dollars flip with probability ≈ |rate × δ| per household (rate 0.24–0.35, |δ| commonly 0.1–1.5). Today's single such anchor passes because 47,031.31's fraction sits far from .50 — the same "residual-position luck" r3-I1 named; over the §5.1 corpus's dozens of mandated high-income cents households, expected undeclared reds ≥ 1. When one fires: the taxcalc Table class is false (no operand consulted the Table — correctly), the provenance class is declared for OTS only, so a **lawful** residual is red with no specced resolution. The same gap covers the QDCGT-with-all-operands-above-ceiling shape (btctax slices on printed operands, `method.rs:87-90`; taxcalc on exact cents). Note the spec's own §6.2(b) sentence claims the predicate "subsumes … TCW-cents" — it does, but only for the oracle the class tuple names. This is the surviving half of r3-I1 failure 3 ("per household per oracle"); r3's fix direction itself under-specified the taxcalc side, and the fold implemented that direction faithfully.

**Fix (one paragraph in §6.4, one line each in §5.1/§12).** Declare the provenance class **per-oracle** — `(O, L16-family, Table_btctax(O's own exact operands) == round_dollar(O L16) AND Table_btctax(reproduced printed operands) ≠ round_dollar(O L16))` for O ∈ {OTS, taxcalc} — keeping the `(taxcalc, Table-operand)` class for the methodology divergence. The composition is automatically clean: **below** the ceiling conjunct 1 *fails* for taxcalc (`Table_btctax` bins, taxcalc doesn't), so the Table class remains the only absorber there and the provenance class cannot over-absorb; **at/above** the ceiling `Table_btctax` *is* the exact schedule (`method.rs:52-55`), so conjunct 1 holds exactly and the class absorbs precisely the lawful cents residual — and a real btctax TCW/schedule bug still fails conjunct 1 and stays red. Taxcalc's exact operands are obtainable (TI at cents is already baked; QD is `e00650`, the net-LTCG term is already hand-derived — `gen_goldens.py:222-230, 177-192`). Liveness: the taxcalc provenance class needs its **own** §5.1 pinned cell (a high-TI cents household steered so the flip occurs — the same checked bake-time steering already specced for the bin-edge cell), recorded in §12 beside the existing obligation.

## Minor

### r4-M1 — §6.2(b)'s "expose the L16 worksheet operands (the ordinary remainder and the preferential slices)" misdescribes what the drivers must extract; as worded it sends the implementer hunting for OTS worksheet-internal output lines that are not known to exist

`Table_btctax` is `qdcgt_line16`, whose signature takes the three **headline** figures (TI, QD, net LTCG) and derives the remainder and slices internally (`method.rs:74-91`). What the drivers must expose is therefore the oracle's exact *leaf* figures — L15 at cents (already parsed and baked), L3a (a driver input, `ots_direct.py:254`), and the Sch-D net-LTCG term (already derived, `:292-294`) — not worksheet internals, which OTS's output is not established to print. Rephrase the parenthetical accordingly, and note the fail-closed property this buys: if an oracle's *internal* worksheet operands ever diverge from what its own leaves imply (an oracle worksheet bug), conjunct 1 fails, nothing absorbs, and §10 triage catches it — which is the correct behavior, worth stating.
