## ★ Adjudication of the rejected r3 I-1 fix: THE AUTHOR IS RIGHT

2. **"Both → None" has no silent-understatement channel — I looked for one and it does not exist.** The hazard would be a v1 population whose stored `true` means a trigger genuinely fired and whose answer is then dropped. But (a) every v1 `hsa_present = true` year **refuses today** (`return_refuse.rs:738`) — none has ever computed; (b) after migration the year refuses as *unanswered* (`HsaActivityUnanswered`, class A) — it still cannot compute; (c) the only path to computation is the filer answering the new four-clause question themselves, and a genuine-activity filer who answers it truthfully is refused unsupported, same as today. 3. **The "mutation is an addition" guard is real, not a rationalization — verified by experiment.** Scratchpad run (serde 1.x, serde_json): a literal v1 blob `{"hsa_present":false}` / `{"hsa_present":true}` into the renamed struct yields `hsa_activity = None` for **both**; adding `#[serde(alias = "hsa_present")]` yields `Some(false)` / `Some(true)` — so the named literal-v1-JSON test asserting `None` for both values **fails on exactly the alias-addition mutation**, and the fixture-build path the spec mandates (serialize current struct, rewrite `"hsa_activity":null → "hsa_present":false`, assert the rewrite landed) works as written. I also checked the third mechanisms: an eager raw-JSON migration is the rejected fix with more machinery; refusing to *load* v1 blobs carrying `true` would brick the year below the level `income answer` can reach (the row cannot load, so nothing can re-ask). The author's mechanism is the best of the available three.

One wording residue only (Nit-1): "permanently refused" overstates — `live_questions` re-asks answered questions with the current value as default (`answer.rs:95-137`, `current_bool`), so a `Some(true)` filer has a recovery path.
