### I-1 — The no-brick property is not held for `MortgageAllUsedToBuyBuildImprove`: dropping it from `income answer` passes the whole suite

**What.** Spec §3.5's per-question property test has three assertions per registry entry; the third is *"assert `income answer` **asks** it"*, and §5 step 4 (rewritten after three consecutive red-step failures, r4 I-3) explicitly includes it: *"Write the … per-question property test red — **all three** of its assertions, including '`income answer` asks it'."* The implementation split assertion 3 into hand-written scenario tests in `answer.rs` that cover **seven of the eight** questions (`a_single_filer_is_asked_the_always_live_declarations…` pins the five always-live; `spouse_questions_appear_exactly_when_a_spouse_does` and `mfs_is_asked_whether_the_spouse_itemizes` pin two more). **No test anywhere exercises `live_questions` on a return where the mortgage question is live** (`grep -rn Mortgage crates/btctax-cli/src/cmd/answer.rs` → no hits; `every_live_question_can_actually_be_answered_and_clears_the_screen` uses an MFJ fixture with no `schedule_a`).

**Why it blocks.** The screen refuses `MixedUseMortgageUnanswered` on every Schedule A carrying mortgage interest; if `income answer` ever stops asking that question, the filer's only exit is re-editing the TOML — the near-brick D-8's recovery story was built to prevent, on the one question whose liveness predicate is non-trivial. Yes, `live_questions` currently derives from the registry by identity — but "true by construction today, held by nothing tomorrow" is the exact convention-not-construction defect P9 indicts, the spec mandated the assertion by name after it went red three revisions running, and steps 6–12 will keep touching this file (step 7 adds the skippable prompts to the same function).

**Evidence (CONFIRMED by mutation).** Added `&& q.id != QuestionId::MortgageAllUsedToBuyBuildImprove` to the filter in `live_questions` (`crates/btctax-cli/src/cmd/answer.rs:95`) → `make check` → **1736 tests run: 1736 passed.** (Restored; verified clean.)

**Fix.** Make assertion 3 registry-derived like the other two: iterate `FORM_QUESTIONS`, build the per-question live scenario (the `scenario_for(q.id)` shape already in `return_refuse::tests`, reproduced or shared), and assert `live_questions(&ri).iter().any(|a| a.declaration_id() == Some(q.id))`. One loop; a dropped or mis-filtered entry then fails a named test for **any** question, including the ones added in steps 6–12.

### I-2 — The §2.6 remedy-chain guards are held by nothing: `income import` silently treating a stale row as absent passes the whole suite

**What.** §2.6 rejected Fable's "treat a stale row as absent" fix **by name**, because it *"silently discards the stale row's computed carryovers — reintroducing, inside P9's own migration policy, the exact class of silent loss P9 exists to close."* §2.6's "Named tests" and §3.5's mutation list therefore require: (a) `income import` over a stale row **refuses** (through the command, not just the error's `Display`); (b) `income clear` + `income import` **succeeds** and the row is v2; (c) **the full-chain restoration test** (seed year N, `--write-carryover` onto N+1, stale both rows, run the whole remedy, assert N+1's carryover equals its pre-stale value); and the mutation *"delete the `clear`-first requirement ⇒ a computed carryover is silently lost ⇒ a named test fails."* **None of these exist.** What exists (`p9_stale_row_refuses`, `return_inputs.rs:238-328`) is excellent as far as it goes — v0/v1/future refuse on `get` **and** `all`, the current version reads, and the `Display` text names all three commands — but it holds only the version gate and the string, not the remedy's behaviour. And no remaining build-order step (6–12: Sch A value, advisories, value-refusals, dead fields, classifier, docs) owns them — the r5 I-3 shape ("specified and scheduled nowhere"), re-run against §2.6.

**Why it blocks.** The mutated `income import` **overwrites** the stale row at v2, skipping the R3-M6 carryover-preservation read entirely — the filer's `Computed` REIT/PTP carryforward vanishes with no error, which §4 R3-M6 established **understates tax**. The guard against it is currently a `?` that nothing asserts; a plausible future "improvement" ("import should just cure the stale row") re-arms it invisibly. The acceptance section says *"Every guard above is mutation-checked, and each mutation is named in §3.5"* — for this guard, that acceptance claim is presently unmet.

**Evidence (CONFIRMED by mutation).** Changed `return_inputs::get(s.conn(), year)?` to `.ok().flatten()` in `import_return_inputs` (`crates/btctax-cli/src/cmd/tax.rs:63`) → `make check` → **1736 tests run: 1736 passed.** (Restored; verified clean.)

**Fix.** Add the three named integration tests from §2.6 to `crates/btctax-cli/tests/tax_report.rs` (the `carryover_write_back_round_trips…` scaffolding already builds the exact N/N+1 vault needed — stale both rows with a direct `UPDATE return_inputs SET schema_version=1`): import-over-stale returns `StaleReturnInputs`; clear+import succeeds and `SELECT schema_version` is 2; the full-chain restoration. Test (a) is the one my mutation kills. ### M-1 — The completeness anchor's comment overclaims what the index round-trip stops

**What.** `questions.rs:201-205` claims *"the index round-trip (r2 M-3) is what stops 'add the match arm, skip the `ALL` element', which would compile green and never be iterated."* It does not: add variant `NewQ`, add arm `NewQ => 8`, touch nothing else — the loop iterates only `ALL`'s 8 existing elements, the new arm never executes, both hardcoded `len() == 8` asserts still see 8, and the suite is green with an enum variant registered nowhere.

**Why not blocking.** The exhaustive `match` still compile-forces a human into this exact test, two lines above the hardcoded counts — the spec's honest limit (§3.3: "a human must EDIT", not "must classify correctly") is met in substance, and the evasion requires editing the anchor while ignoring what it says. But the comment asserts a force the mechanism does not have — the same overclaim shape r3 I-5 was graded for in the spec.

**Evidence.** `crates/btctax-core/src/tax/questions.rs:201-228`; the walk-through above (verifiable by adding a variant — not run, as it perturbs `QuestionId::ALL` consumers).

**Fix.** Correct the comment to what actually holds ("a new variant forces a human edit here; the hardcoded 8s are the tripwire they land next to"), or harden: give `QuestionId` a `const fn index(self) -> usize` (the match moves there) and assert both `ALL[id.index()] == id` for every element **and** `ALL.len() == QuestionId::MortgageAllUsedToBuyBuildImprove.index() + 1` — then a new last variant breaks the length assert too.
