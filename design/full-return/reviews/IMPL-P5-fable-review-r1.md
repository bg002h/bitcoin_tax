## Critical

**C1 — LIMITATIONS.md:47–54 "Forms — computed vs. filled": the "Filled as an official IRS PDF" list misstates what `export-irs-pdf` produces for a full return; following the doc can file an understated Schedule D.**
The doc claims *"**Filled as an official IRS PDF** (`export-irs-pdf`): **1040 · Schedule D · Schedule SE · Form 8949 · Form 8283**"* and instructs transcription only for "the remaining schedules and forms (1, 2, 3, A, B, C, and Forms 8959, 8960, 8995)" — i.e., it affirmatively tells the filer the five listed forms are done. Verified against source at `00a594d`, the export is the **crypto-pipeline** fill with no full-return awareness:
- `crates/btctax-cli/src/cmd/admin.rs:199+` (`export_irs_pdf`) reads only the ledger projection — it never consults `ReturnInputs` and has no guard/warning for a full-return year.
- `crates/btctax-forms/src/schedule_d.rs:51–110` fills **only lines 3/7/10/15/16** from crypto totals — never **line 13** (1099-DIV box-2a capital-gain distributions) and never **lines 6/14** (capital-loss carryovers), both of which ARE inside the computed absolute L7 (`return_1040.rs:292–300` `capital_gain_line7` → `capital_net(ri, …)`).
- `crates/btctax-forms/src/form1040.rs:71` (`fill_form_1040_capgains`) fills only the capital-gains cluster (line 7 + digital-asset question) — not wages/AGI/tax/withholding.

Concrete failing scenario (uses only income the doc itself lists as supported): a household with crypto disposals **and 1099-DIV box 2a**. `report --tax-year 2024` correctly includes the distributions in L7; `export-irs-pdf` writes a complete-looking Schedule D with line 13 blank and line 16 = crypto-only; the doc says Schedule D is filled (it is excluded from the transcription list); the filer mails it → **income understated on the filed return**. With a carryforward-in the filled line 16 errs the other way. Either way the filled form contradicts the computed return the doc says it embodies. Fix: state that today's `export-irs-pdf` fills the **crypto slice only** (8949/Sch D from the ledger, Sch SE, 8283, and the 1040 capital-gain line + DA question), that these do **not** carry the full return's non-crypto capital items, and that until the P6 fillers land **every** form of a full return must be transcribed from the report — and/or make `export-irs-pdf` warn or refuse for a `ReturnInputs` year (worth its own follow-up regardless of the doc fix).

## Important

**I1 — LIMITATIONS.md:7–8: "Every full return it produces is a DRAFT (watermarked) and requires your attestation before export" — not true of the shipped binary.**
The watermark + attestation gate is **pseudo-mode-only**: `cmd/admin.rs:211–223` (`let watermarked = state.pseudo_active();` — attestation required and DRAFT stamped only then). A real-ledger export needs no attestation and carries no watermark, and no full-return PDF exists at all yet (SPEC §9 / plan phase 6 list "DRAFT/attest gate" as **P6** scope). The safety doc claims a control that doesn't exist — the same control C1 relies on being absent. Reword to present-tense truth (pseudo-only today; the always-on full-return gate lands with the P6 fillers).

**I2 — LIMITATIONS.md:66 "Each fires a loud advisory on the report" — false for 2 of the 4 OMISSIONS rows; SPEC's "(advisory, not refused)" promise unmet.**
`Advisory` (`crates/btctax-core/src/tax/advisories.rs:26–48`) has exactly 5 variants; grep of core + render confirms **no advisory exists** for the education/dependent-care/saver's/energy/adoption-credits row or the direct-deposit row. SPEC §1.2 ("education, dependent-care, saver's, energy, adoption credits — omitted conservatively (**advisory**, not refused)") and §9.2 ("(i) favorable-only OMISSIONS (**advisory**…) — …, direct-deposit") both commit to advisories for these. Failing scenario: a filer eligible for the saver's credit reads the doc's "each fires a loud advisory," sees none on their report, and concludes no omission applies. Fix either direction: add a static "other credits not computed / refund by paper check" line to the full-return block, or scope the doc sentence to the rows that do fire — and reconcile the SPEC.

**I3 — `advisories.rs:19–22,461` EIC ceiling: the cited figure is wrong and the constant UNDER-fires for MFJ — violating its own stated safety direction.**
The comment reads *"Deliberately a round over-estimate of the TY2024 maximum EIC AGI (≈ $59,899 for MFJ with 3+ children) … over-firing is the safe direction."* Per Rev. Proc. 2023-34, **$59,899 is the Single/HoH/QSS 3-child limit; MFJ is $66,819** (2 children: $62,688) — so `EIC_ADVISORY_AGI_CEILING = dec!(60000)` is an *under*-estimate for MFJ with 2+ children. Reproduced on the built binary: MFJ, 3 dependents, $63,000 earned AGI → the report printed the CTC advisory but **no EIC advisory**, for a household squarely inside EIC territory (max credit $7,830). The design intent (over-fire = safe) is stated in the code and in the doc's table row; the implementation misses a real band. Fix: raise the ceiling to ≥ $67,000 (a round $70,000 keeps it safe for several years) and correct the comment; add a KAT at MFJ/$63k.

**I4 — `crates/btctax-cli/src/main.rs:417` `include_str!("../../../LIMITATIONS.md")` breaks `cargo publish` for btctax-cli.**
LIMITATIONS.md lives outside the package root and is **not in the .crate tarball** (verified: `cargo package -p btctax-cli --list` does not contain it), so the publish verification build of the packaged crate cannot compile. Every workspace crate is routinely published (v0.2.0 → 0.5.0 lineage); this fails loudly at the next release.
