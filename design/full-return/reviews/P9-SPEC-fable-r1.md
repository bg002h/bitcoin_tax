## Important

### I-1 — The refusal text cannot be "derived from q.prompt", and step 7 asserts a false premise

§2.1 sketches `refuse(q.unanswered.clone(), /* text derived from q.prompt */)`. Three problems:

1. **It breaks a named acceptance test or loses the remedy.** `tax_report.rs:1792` asserts the D-8 refusal `msg.contains("income answer")`. The prompt (`answer.rs:42`) does not contain it. If the derivation is prompt-only, the P8a acceptance test goes red; if the author improvises a remedy suffix, the spec never specced the one string a refusal is required to carry ("a refusal with no exit is just a brick with better prose" — the project's own doctrine, and `LIMITATIONS.md:140` documents it).
2. **It loses question-specific context.** The Schedule B text explains *why the question is live* ("Schedule B is required (interest/dividends > $1,500 or a foreign account)…", `return_refuse.rs:520–522`); the MFS text cites §63(c)(6). A prompt-derived text drops both.
3. **Step 7 is factually wrong as written**: "verify the four refusal texts **still** name `btctax income answer`" — only the two D-8 texts do today (`return_refuse.rs:585, :593`); Schedule B and MFS name TOML keys instead. Making all four name `income answer` is a *change* (arguably an improvement — `income answer` does ask those questions), which contradicts the same bullet's "no user-visible change expected."

**Fix:** the entry carries the full detail — e.g. `unanswered_detail: &'static str` (or `why: &'static str` composed with a standard `— run btctax income answer` suffix, specced as such). Rewrite step 7 to say the Schedule B/MFS texts *gain* the remedy and that this is deliberate.

### I-2 — `ReturnHeader::build`'s new failure mode has no error type, and the naive one misreports

`build` returns `Result<Self, SsnError>` (`packet.rs:275`), and `SsnError` (`packet.rs:92–99`, `Copy`, three SSN-shaped variants) cannot represent "a live question is unanswered." The spec says only "it gains: … `Err`." Whatever variant is shoehorned in, the production caller maps every build error to *"the {year} return cannot be printed: {e} — **fix the identity** and re-run"* (`admin.rs:477–480`) — an unanswered dependent question reported as an identity problem, naming no remedy. Signature widening also propagates through `assemble_printed_return` (`packet.rs:407`) and touches `testonly.rs:286`, `extract_lines.rs:262`, and four `full_return_forms.rs` call sites, whose fixtures must now answer every live question or panic with an SSN-flavored `expect`. **Fix:** spec the error shape (e.g. `enum HeaderError { Ssn(SsnError), Unanswered(QuestionId) }`), the `Display` for the unanswered arm (prompt + `income answer` remedy, consistent with I-1), and the caller mapping.

### I-3 — The classifier's guarantee stops one struct-level short of where the bools actually live

The no-`..`, no-`_` destructure covers `ReturnInputs` + `HouseholdHeader` only. But of the fields that need classifying **today**, most are nested: `Person.blind` (`return_inputs.rs:132`), `Person.ssn_valid_for_employment` (`:129`, also on `Dependent` `:143`), `W2.box13_retirement_plan` (`:58`), `ScheduleAInputs.salt_use_sales_tax` (`:288`), `Schedule1Inputs.hsa_present` (`:321`). A top-two-level classifier classifies **none of them**, and a new `bool` on `Person` or `Schedule1Inputs` tomorrow compiles clean — the guarantee is narrower than §1's "closes the whole class" and §2.3's "the part that actually stops the next D-8," which is precisely the sin the spec charges to `first_negative_amount`'s `header: _`. Note `first_negative_amount` itself already recurses into `W2`, `ScheduleAInputs`, `Schedule1Inputs`, `ScheduleCInputs`, `Payments`, `QbiInputs` (`return_refuse.rs:223–481`) — the classifier must recurse at least as deep (or share that destructure). Also record the class boundary honestly: defaulted **enums** that print or attribute escape a bool-only net — `AccountingMethod` (the already-filed accrual Important, FOLLOWUPS `→ P8`) and `Owner` (`#[default] Taxpayer` silently attributes a spouse's W-2 on MFJ, touching per-earner §31(b)/§402(g) buckets) are the standing examples.

### I-4 — "Drop an entry from `FORM_QUESTIONS` → a named test fails" has no mechanism, and the obvious implementation is vacuous

The §2.2 property test **iterates the registry** — drop an entry and its scenario silently vanishes; the suite stays green. Simultaneously, §2.2 "replaces per-question tests forever," i.e. invites deleting exactly the hand-written tests that would have caught the drop (`an_unanswered_dependent_flag_refuses`, `return_refuse.rs:820`; the Schedule B/MFS assertions at `:1311–1325`). This is the vacuous-fixture shape P8a I2 just demonstrated, rebuilt into the anti-vacuity machinery itself. **Fix:** spec the completeness anchor — a `QuestionId::ALL` whose exhaustiveness is compiler-held by a `match` (or the classifier's field→`QuestionId` mapping), with a test asserting every variant has exactly one registry entry. Then the drop-mutation genuinely fails a named test.

### I-5 — The census misses a latent instance of the I7/D-8 print shape: the dual-status-alien half of the checkbox btctax already prints

The 1040's third box is "**Spouse itemizes on a separate return or you were a dual-status alien**" (`packet.rs:251`, `map.rs:260`). btctax prints it from the MFS coupling alone (`packet.rs:325–326`) — the dual-status half is silently answered "No" for every filer, with **no field, no question, no refusal, and no LIMITATIONS entry** (grep: zero hits for resident/alien anywhere but the checkbox's own name). A dual-status alien's standard deduction is barred (§63(c)(6)(B)); btctax's `ItemizeElection::Auto` grants it — a silent understatement plus an unaffirmed checkbox, for a population (first-year visa holders with W-2s and crypto) squarely inside the app's archetype. This meets the spec's own criterion exactly, and it is the identical shape to SPEC r1 I7 ("named on day one"). **Fix:** either a sixth registry entry (a new `Option<bool>`, with the migration-free advantage that `None` on old rows is honest) or an explicit written exemption + LIMITATIONS entry + follow-up with an owning phase. A spec whose acceptance says "all five existing questions" while claiming class closure must either close or disclaim this one.

### I-6 — `income answer` (as re-derived here) manufactures a facially incomplete Schedule B: 7a "Yes" with 7b blank

Answering "y" to `ForeignAccounts` via `income answer` sets `foreign_accounts = Some(true)` and nothing captures `foreign_country_names` — the command has no prompt for it, no screen guards it (`return_refuse.rs:220` waves it with `_`), and the filler writes 7b only when non-empty (`schedule_b.rs:176`). Result: a filed Schedule B with 7a checked "Yes" and line 7b ("If 'Yes', enter the name of the foreign country") blank — the same required-text-line defect class as `ScheduleCNoBusinessDescription`, which was graded Important (P7 r2 I2) on identical reasoning. This predates the spec, but the spec re-derives the very command that creates the state, and its "value-dependent refusals stay hand-written" bullet implicitly asserts that hand-written set is complete. **Fix:** a value-dependent refusal (`Some(true) && foreign_country_names.trim().is_empty()`), or a country prompt in `answer` when 7a is answered yes — specced here or filed with an owning phase before this spec ships.

## Minor

- **M-1** — §2.1 says `Question`/`live_questions`/`current_bool`/`set_bool` "all delete," but step 4 keeps the DOB machinery, which lives in the same enum with `is_date`/`is_skippable`/prompts (`answer.rs:31–75, 101–104, 121–127, 144–154`). The enum cannot wholly delete; say "shrinks to the DOB residue" or the implementor will fight the spec.
- **M-2** — The registry loop's position in `screen_inputs` is unspecified. It must sit after the integrity gates (negative money, malformed SSN), and cross-field refusal precedence shifts for multi-defect returns (e.g. `foreign_trust = Some(true)` + unanswered 7a on a >$1,500 year currently reports `ForeignTrust` first, `return_refuse.rs:508` vs `:517`; the loop would reverse it). State the position and that precedence is not contract.
- **M-3** — `Mfj || spouse.is_some()` keeps the spouse-dependent question live on MFS/QSS returns carrying a spouse `Person`, though the form's box is joint-only per the 1040 instructions, and `DependentSpouseUnsupported`'s joint-std rationale (`return_refuse.rs:84–85`) does not apply to MFS. Conservative and pre-existing; record it as a deliberate over-ask.
- **M-4** — The exemption criterion "absence must be conservative **and advised**" does not fit the population it must classify: `blind = false` is conservative but nothing advises it; `hsa_present` is a declaration-of-existence whose absence is only conservative because W-2 code W falls outside the box-12 allowlist (`return_refuse.rs:28`) — that reasoning belongs in its written entry; `presidential_fund_*` has no tax direction at all. The criterion needs the extra categories or every honest entry fails its own test.
- **M-5** — **Q6, argued out: keep the widening; it is coherent.** The question is about the real-world spouse an MFJ return has by definition, not the captured `Person`; the flag lives on `HouseholdHeader`, not inside `Person`, so storing the answer spouse-less is not a data-model lie; and the answer is compute-relevant (`Some(true)` → refuse; `None` → screen; `Some(false)` → full joint std). The alternative — refusing MFJ-without-spouse-Person at `screen_inputs` — would contradict the project's own seam ("an uncaptured SSN does not block the report," `return_refuse.rs:997–1005`): identity gaps gate the **packet**, not compute. But that is where the residue lands: `ReturnHeader::build` today builds an MFJ packet with no spouse name/SSN/signature block at all (`packet.rs:277–287` — spouse `None` is simply skipped), and the existing follow-up covers only MFS (`p6-r1-m2-mfs-without-spouse → P6.7`, FOLLOWUPS:479). Since step 5 is already inside `build`, add the one-line MFJ-requires-spouse-identity refusal there (via I-2's error type), or file the MFJ half with an owning phase. The acceptance sentence "the MFJ-with-no-spouse-Person hole closes as a consequence" is accurate for I1-as-filed but should name this surviving half.

---

**Bottom line:** build it — the registry, the derivations, the widened predicate, and the witness rejection all survive attack, and the const-registry mechanics are verified real. But the spec ships text and error boundaries it never specifies (I-1, I-2), an anti-vacuity claim its own test design defeats (I-4), a classifier one level shallower than its rhetoric (I-3), and a census that neither closes nor disclaims the one unregistered checkbox btctax already answers silently on every filed 1040 (I-5), while re-deriving a command that can file a 7a-Yes/7b-blank Schedule B (I-6). All six are bounded, spec-level fixes; none undermines the architecture.
