### I-1 — §2.6's refusal names `btctax income import` as the remedy, but `income import` itself reads the stale row through the refusing boundary and ERRORS — the named exit fails on every row it exists to cure

**What.** `import_return_inputs` (`crates/btctax-cli/src/cmd/tax.rs:63`) begins with `if let Some(existing) = return_inputs::get(s.conn(), year)?` — the carryover-preservation read (§4 R3-M6). The same circle binds `income answer` (`answer.rs:191`, get-then-set) and `report --write-carryover` (`tax.rs:412/:426`). **Why it blocks.** §4's acceptance — *"Stale stored rows REFUSE … naming `btctax income import` as the remedy"* — is an unmet guarantee as specced: the named remedy provably fails on the population the refusal targets. r2 C-1 established that a refusal whose stated exit doesn't work is a blocking defect even when the data is expendable.

**Evidence.** `cmd/tax.rs:63` (`return_inputs::get(...)?` inside `import_return_inputs`); `return_inputs.rs:70-82` (get routes through `row_to_inputs`); `answer.rs:191`; `tax.rs:412/:426`; §2.6/§4 of the artifact.

**Fix (spec-level, three sentences in §2.6 + one in step 2).** `income import` treats a stale existing row **as absent** for the carryover-preservation read (catch `StaleReturnInputs` from that one call and proceed as if no row existed — the preserved carryovers are themselves stale, and the whole point of the command is to replace the row; note this in the R3-M6 comment). `income answer`'s and `write-carryover`'s stale errors keep the re-import remedy, which now works. Add a named test: import over a v1 row **succeeds** and the row is v2 afterward; delete the catch ⇒ it fails.
