## Findings

### IMPORTANT

**I-1. "Same corpora, shared fixtures" — the shared fixtures do not exist.** §4 step 1 says drivers seed "the *same corpus* the corresponding examples journey uses … **via the existing CLI/test fixtures**", and §7 says "Same corpora as the examples journeys (**shared fixtures**) so parity is structural." False as written: all nine corpora are **private consts inside `crates/xtask/src/examples.rs:203-291`** (`J1_CSV` … `J9_CSV`), a bin-only, `publish = false` crate that neither TUI crate depends on (and must not — xtask depends on btctax-cli, not the reverse). They are embedded as CRLF strings precisely because committed `.csv` files get force-LF'd (examples.rs:201-202), so "just commit the files and share them" is also off the table without `.gitattributes` surgery. J-parity therefore requires a refactor the spec doesn't own: hoist the corpora to a shared home — `btctax-cli` is the natural one (xtask, btctax-tui, and btctax-tui-edit all already depend on it; precedent for testonly exposure exists in `btctax-forms` per xtask/Cargo.toml:18-20) — or duplicate them and forfeit "parity is structural". The spec must state which, and the plan must carry the step. (The import path itself is fine: `btctax_cli::cmd::import::run(vault, pp, files)` is pub — `crates/btctax-cli/src/cmd/import.rs:10` — so a driver can write the corpus to a tempdir and run the real adapter ingest.)

**I-2. The single-driver "editor flow → then switch to the viewer for the result tabs" sketch is not implementable across the crate boundary as written.** §4 step 2 describes one driver capturing editor states "then switch[ing] to the viewer for the result tabs". But the viewer's `App` is deliberately `pub(crate)` — `crates/btctax-tui/src/app.rs:137-139`: "Kept `pub(crate)` so the viewer's internal surface does not leak into the editor crate" — as are `handle_key` (lib.rs:164) and the full-frame `draw` path and per-tab `draw` wrappers (e.g. tabs/holdings.rs:169). btctax-tui-edit is bin-only (no `[lib]` target — `crates/btctax-tui-edit/Cargo.toml`), and xtask depends on neither TUI crate. Consequences the spec doesn't surface:
- Editor frames can ONLY be captured from `#[cfg(test)]` modules inside btctax-tui-edit's `src/` (as today); viewer full-chrome frames (tab bar, export modal, what-if overlay) can ONLY be captured inside btctax-tui. The per-tab `pub fn render(frame, area, &Snapshot, year, …)` functions (tabs/holdings.rs:71, tabs/tax.rs:22) DO let the editor crate capture viewer **content panes** (Snapshot is pub, and the editor already calls `btctax_tui::unlock::build_snapshot`), but not chrome/modals/overlay.
- So "editor mutation → viewer result" for one journey either (a) splits across two crates' emit tests, with the second crate re-seeding the same corpus and **replaying the mutation** via btctax-cli calls — feasible (btctax-tui depends on btctax-cli; unlock.rs uses `Session`) but introducing a convergence-by-construction discipline between two halves of one journey that the spec must acknowledge and pin (a shared per-journey fixture fn, owned once — ties into I-1's shared-fixtures home); or (b) a deliberate testonly visibility relaxation in btctax-tui reversing a documented encapsulation decision — which needs an explicit spec-level call, not a plan-level drift.
- §8's alternative "or an xtask-driven harness" is infeasible for editor frames as the code stands and should be struck or explicitly re-scoped to assembly-only (xtask can assemble committed goldens; it can never capture them).

**I-3. Determinism hole: the price-cache layer.** §7's purity claim lists clock + vault path + geometry, but every snapshot built through the real unlock path layers the **developer's live local price cache**: `crates/btctax-tui/src/unlock.rs:171` → `LayeredPrices::load_with_cache(btctax_cli::price_cache::default_cache_path().as_deref())` — deliberately "the real cache path (never `None`)" (unlock.rs comment, R0-I1) — and `default_cache_path()` honors `BTCTAX_PRICE_CACHE` else falls back to `dirs::data_dir()/btctax/price_cache.csv` (`crates/btctax-cli/src/price_cache.rs:19-24`). Today's two editor goldens are latently safe only because no captured cell renders a cache-resolvable price. The walkthrough's frames are exactly the ones that aren't: J4 income FMV, J7's FMV field, J5's what-if prices. On any machine with a populated cache (this project ships `btctax-update-prices`), regen would differ from CI. The xtask examples generator pins `BTCTAX_PRICE_CACHE` to a nonexistent file for precisely this reason (examples.rs:96, `.env("BTCTAX_PRICE_CACHE", cwd.join("no-such-price-cache.csv"))`). §7 needs the same pin for the drivers (env-var set inside the gate/emit tests is safe under nextest's process-per-test model, which `make check` uses; note plain `cargo test` in-process threading as a caveat).

**I-4. §5's render pipeline is presented as reuse, but the interleaved renderer does not exist.** Today there are two disjoint pipelines: `man-wrap.awk` renders Markdown **prose** (examples.md → btctax-examples.pdf, Makefile:58-64), and `tui-wrap.awk` renders one golden **`.txt`** per `.SH` section with no prose (Makefile:70-83 loops `docs/examples-tui/*.txt`). The walkthrough needs prose+captions interleaved with colorized frames in one groff stream, and the "committed assembled Markdown" cannot itself carry the colorization — `tui-wrap.awk` consumes the glyphs+style-runs `.txt` format, not Markdown, and `man-wrap.awk` knows nothing of style runs. So `make tui-walkthrough` requires new glue: either an ordering manifest the Makefile walks (running man-wrap.awk on prose chunks and tui-wrap.awk on frame files, concatenated in journey order), or the xtask assembler emitting roff directly. Modest work — but the spec must say what the committed assembled artifact IS (prose + frame references, vs prose + inlined uncolored grids) because both the byte-gate story ("byte-gated like examples.md") and the render pipeline hang off that decision. As written, §5 implies the existing pipeline suffices; it doesn't.

### MINOR

**M-1. J5 hedge is resolvable now, and the journey table under-claims two journeys.** The optimize-accept flow is surfaced (`z`, main.rs:452/9248) — drop "(if surfaced)". J2 can capture the editor's set-donation-details flow (`d` → `open_set_donation_details_flow`, main.rs:~433) rather than being "thin–medium viewer-only"; J5's narrated "profile, FIFO election" both have real editor surfaces (`p` profile form, `e` method-election flow, main.rs:426/453). The thin-journey concern is genuinely milder than §11 fears — J1 is the only truly viewer-only journey, and its four frames (Holdings → Disposals → Tax → export modal) are a legitimate "read your results" chapter. The PoC + owner-approval gate (§9) adequately covers the "is nine worth it" question; no stronger language needed.

**M-2. Caption/narration ownership is split three ways.** As designed, frames+drivers live in two TUI crates' test modules while captions and narrated preambles live in the xtask assembler — one journey's content spans three crates. Recommend the spec name a single owner for per-frame caption text (the assembler's manifest is the natural place, with drivers emitting only frames named by the manifest), so a caption edit can never require touching a TUI crate.

**M-3. Frame-list stability.** Multi-step flows mean many goldens (§6 sketches ~4-8 frames × 9 journeys ≈ 40-70 goldens, roughly 10-20× today's four). Every real UI change reds them all — §11 accepts this, but the emit story is now TWO `#[ignore]` tests in two crates plus an xtask regen; recommend a single documented `make`-target regen sequence so the "one-command refresh" claim (§11) stays true.

### NIT

**N-1.** §4 cites "the viewer's `render_*` capture helpers in `tabs/tests.rs`" — correct, but they are inside `#[cfg(test)] mod tests;` (tabs/mod.rs:15), i.e. not reusable outside the crate's own test build; the spec text reads as if they're ambient machinery. Subsumed by I-2's decision.

**N-2.** §5's geometry string matches the Makefile exactly (`-dpaper=letterl -P-pletterl -rLL=10i -rPO=0.4i`, Makefile:61/80) — no drift, just noting it was verified.

## VERDICT

**NOT GREEN — 0 Critical / 4 Important.** The spec's core bet is sound and proven by existing code: `edit-classify-confirm-modal` already demonstrates the exact seed-vault → scripted-`KeyEvent`s → pinned-clock/path → `TestBackend(120,40)` → `to_golden` pattern, every claimed journey surface exists (including J5's optimize-accept), and the gating story mirrors the working examples/examples-tui pipeline. But before this goes to a plan, the spec must: (I-1) own the corpora-hoisting refactor instead of claiming shared fixtures exist; (I-2) decide the cross-crate capture architecture (split drivers with replayed mutations vs a testonly visibility relaxation) with eyes open on the `pub(crate)` encapsulation it touches, and strike the infeasible xtask-capture alternative; (I-3) add the price-cache pin to §7's determinism list; (I-4) specify the committed assembled artifact's format and the new prose/frame interleaving glue that `make tui-walkthrough` actually requires. All four are spec-text amendments plus plan-level work items — none invalidates the feature.
