## New findings

**Important NEW-1 — the NF-2 arm's condition is an untested guard: BOTH single-conjunct mutants survive the full suite (empirically verified, 2075/2075 PASS each).** The condition is `!had_unconservable_before && any(SafeHarborUnconservable)` (main.rs:4948-4954). Killed: whole-arm-off and guard-polarity-flip (both by the multilot KAT). Surviving:

- *Mutant A — guard dropped* (condition → `any(...)`): full workspace suite passes. Failure scenario the guard exists to prevent, now unheld: a filer with a PRE-EXISTING broken allocation (e.g. attests a wrong Σ-basis → baseline `SafeHarborUnconservable`) saves a perfectly valid, unrelated selection → status falsely asserts "BUT it broke your safe-harbor allocation's conservation … Void this selection (press 'v') to restore it" — voiding that selection would NOT restore anything. The `had_unconservable_before` mechanism is the item's own headline device, and nothing in the suite discriminates it.
- *Mutant B — any-conjunct dropped* (condition → `!had_unconservable_before`): full workspace suite passes. Failure scenario: EVERY clean first select-lots save in the app reports "the allocation is now inert and the year will not compute" — a false year-gating alarm on the happy path — and the suite stays green. Root cause: the arm-3 message deliberately opens with the same "Lot selection recorded" prefix as clean arm-4, and the only two clean-save status asserts are bare `contains("Lot selection recorded")` (main.rs:21724, 21983), which match EITHER arm; no walkthrough golden pins the post-save status (`docs/examples-tui-walkthrough/j9/01-select-lots.txt` has zero occurrences).

Under this project's explicit standard (untested-guard pattern: "a fix isn't done until the mutation dies") and the r2/r3 precedent of treating test anchoring as Important, SL-r2-a is not done. The fix is small: (1) tighten one existing clean-save assert to pin arm-4 distinctly — `contains("contemporaneity")` or `!contains("BUT it broke")` — kills Mutant B; (2) add a pre-existing-break KAT — seed a NON-conserving allocation (attest a wrong Σ-basis; baseline asserts `SafeHarborUnconservable` present), save a feasible pick, assert the status does NOT carry the "BUT it broke" attribution — kills Mutant A. Consider also de-aliasing the arm-3 prefix so future substring asserts can't conflate the arms.

**Minor NEW-2 — the guard is any-allocation boolean-scoped, giving a masking false-negative in a two-allocation ledger.** The resolver checks every non-voided allocation and can carry an Unconservable on inert alloc-A while alloc-B is the single effective Path-B governor (resolve.rs loop + :1293-1313 — only multiple EFFECTIVES conflict, so broken-A + effective-B is a legal state). A pick that newly breaks B is then masked by A's pre-existing blocker (`had_unconservable_before == true`) → clean arm-4 message while the GOVERNING allocation just went inert. Harm is bounded — A's Hard blocker already gated the year — which is why this is Minor, not Important. A set-delta on unconservable-carrying allocation ids (before vs after) instead of a boolean would make the attribution exact. File with an owner alongside the NEW-1 tests.

**Nit** — the M-1 error message renders `remaining_sat` twice in one sentence ("only 30000 sat available; reduce it to at most 30000"); one clause would do.

**Nit** — NF-1's ActualPosition nuance (above): record in the FOLLOWUPS scope note that any future composition-drift decision should be scoped to `AllocMethod::ActualPosition`, where the attestation semantically claims to mirror actual position.
