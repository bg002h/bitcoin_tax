### [I1] IMPORTANT — optimize-accept's per-disposal "Δtax" column/modal field is not backed by the data model
**Where:** SPEC lines 129–130 ("Table: Date | Wallet | **Δtax** | Persistability | disposal EventId") and 136–137 ("Modal shows … the proposed `LotSelection` picks …, **Δtax**, basis label …").
**Ground truth:** `OptimizeProposal` (`optimize.rs:85-103`) carries exactly one tax-saving figure — `delta: Usd` — and it is the **whole-year** optimized−baseline delta. `DisposalProposal` (`optimize.rs:49-57`) has **no** per-disposal tax/delta field, and `optimize_year` never computes one. The CLI renderer confirms this: `render_optimize_proposal` prints the three whole-year dollar lines (`baseline_tax` / `optimized_tax` / `delta`, `render.rs:1602-1616`) and, per disposal, only `disposal · date · wallet · compliance-tag · proposed-picks` — **no per-disposal dollar amount** (`render.rs:1617-1637`).
**Why it gates:** The saving is the headline of this flow, and the spec asks for it as a per-row column and a per-disposal modal value that does not exist and is not well-defined per disposal — the optimizer minimizes tax *holistically* across contended same-wallet disposals (`optimize.rs:752-839`), so the joint `delta` cannot be cleanly attributed to a single disposal, and accepting only some disposals will not realize any per-disposal figure. An implementer following the spec literally will either stall or (worse) fill the column with the whole-year `delta` on every row, or invent an un-specified single-disposal rescore — an affirmatively misleading number, exactly the class of dishonesty the surrounding invariants forbid (the approximate-banner / "never present optimized tax as the optimum" discipline, `optimize.rs:93-101`).
**Fix (cheap — pick one and pin it):** (a) Drop the per-row Δtax column; show the whole-year `OptimizeProposal.delta` once, as a flow-level banner labeled "expected year Δtax if the full proposal is accepted (delta ≤ 0)", alongside the existing `approximate` banner. Keep per-row = `date | wallet | persistability | disposal id` (+ optionally a "changes N lots" hint). Or (b) if a per-disposal figure is genuinely wanted, specify its exact computation (e.g. baseline vs baseline-with-only-this-disposal's-proposed-pick rescored through `compute_tax_year`) AND state the caveat that per-disposal figures need not sum to the joint `delta`. Either resolves the ambiguity; (a) is simpler and matches the CLI.

### [M1] MINOR — `tax_date(now)` has the wrong arity in the opener recipe
**Where:** SPEC line 114 — "`proposal_made` ← `tax_date(now)`".
**Ground truth:** `pub fn tax_date(utc: OffsetDateTime, tz: UtcOffset) -> TaxDate` (`conventions.rs:52`); the CLI calls `tax_date(now, UtcOffset::UTC)` (`optimize.rs:47`, `181`).
**Why it matters:** A literal implementation is a compile error; more importantly the tz seam is load-bearing (decisions are UTC, and the same `made` must be used for both the persistability recompute and the attestation's `attested_at`).
**Fix:** `proposal_made ← tax_date(now, UtcOffset::UTC)`.

### [M2] MINOR — `map_opt_err` is `pub(crate)` (not reusable from the TUI), and the error-arm list is off for `optimize_year`
**Where:** SPEC lines 118–120 — "Map optimizer errors → status + no-open via `map_opt_err`'s cases: `PreTransitionYear`, `YearNotComputable`, `NoDisposals`, `NoLots`."
**Ground truth:** `map_opt_err` is `pub(crate) fn` in `btctax-cli` (`cmd/optimize.rs:69`) — **not** callable from `btctax-tui-edit`. And `optimize_year` can only return `PreTransitionYear` / `YearNotComputable` / `NoDisposals` (`optimize.rs:723-750`); `NoLots` and `Evaluate` are **Mode-2 (`consult_sale`) only** — unreachable here.
**Why it matters:** The spec presents the free-standing assembly as fully equivalent to the helper; but the free-standing path cannot reuse `map_opt_err` and must hand-roll the `OptimizeError` → status mapping over the **3** reachable arms (not 4). Following the spec's 4-arm list yields a dead `NoLots` arm and omits nothing important, but it signals a wrong mental model of what `optimize_year` returns.
**Fix:** Prefer the recommended `Session::optimize_proposal(&self, year, now) -> Result<OptimizeProposal, CliError>` helper that applies `.map_err(map_opt_err)` internally (returns a pre-formatted `CliError` the TUI renders to status). If the free-standing path is chosen, enumerate the 3 reachable arms and drop `NoLots`.

### [M3] MINOR — behavior when the post-filter proposal list is empty is unspecified
**Where:** D4 pre-filter (SPEC lines 121–126); opener step (129–134).
**Gap:** After (no-change ∪ `ForbiddenBroker2027` ∪ has-live-`LotSelection`) filtering, the candidate list can be **empty** (e.g. everything already optimal). The spec does not state the opener's behavior in that case. Every other list-flow pins this (e.g. void's R0-M8 "empty filtered list → status, flow NOT opened", `main.rs:2491`).
**Fix:** Add: "empty post-filter list → set a status (e.g. `\"No optimizer-improvable disposals to accept.\"`) and do NOT open the flow," mirroring the void empty-list convention.

### [N1] NIT — opener mixes a fresh event load with the (rebuilt) Snapshot for `profiles`/`tables`
**Where:** SPEC lines 112–114 — `events,config ← session.load_events_and_project()` (fresh) but `tables ← snap.tables`, `profile ← snap.profiles.get(&year)` (from the app Snapshot).
**Observation:** This is correct **given** the editor's rebuild-Snapshot-after-every-successful-save invariant (tables are static; `snap.profiles` is current post-save). Not a defect. But the recommended `Session::optimize_proposal` helper reads `s.tax_profile(year)` fresh (as `run`/`accept` do, `optimize.rs:43/178`) and avoids any fresh-vs-snapshot coupling — prefer it for consistency, and if the free-standing path is used, read the profile fresh via `session.tax_profile(year)` rather than `snap.profiles`.
